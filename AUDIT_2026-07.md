# Code, structure & security audit — July 2026

Full review of the repository against `470_Project_Group5 (4).pdf` (the SRS).
Findings are ordered by severity. Each says what was wrong, why it mattered,
and whether it is now fixed.

The headline: **`SRS_FEATURE_STATUS.md` significantly overstated completion.**
Five features it marked **Done** did not meet their own Section 7 acceptance
criteria. That file has been corrected.

---

## 1. Security

### 1.1 Broken access control — claim tagging (fixed)

`POST /articles/{id}/claims` checked that the caller was *a* journalist, never
that they were *this article's* journalist.

Any signed-in journalist could attach claims to a competitor's article. Since
claims drive `false_claims`, which drives both the Corruption Factor (formula
2) and the Rank Score (formula 1), this was a direct route to tanking a rival's
public score: tag their article with claims, let them be voted false, watch
their node redden and their rank drop.

**Fixed** in `internal/claims/handler.go` — ownership is verified, and claims
on retracted articles are refused.

### 1.2 Broken access control — appeals (fixed)

`POST /appeals` took `articleId` from the request body and never checked
ownership either. Any journalist could file an appeal against any article,
pinning a competitor's node into the pulsing amber "under dispute" state
(FR-9) indefinitely, since nothing limited how many appeals could be open.

**Fixed** in `internal/articles/handler.go` — ownership verified, one active
appeal per article enforced by a partial unique index.

### 1.3 `credential_verified` defaulted to `true` (fixed)

```sql
credential_verified BOOLEAN NOT NULL DEFAULT true
```

NFR-6 (Sybil resistance) is a security gate, and it was failing **open**. Any
auditor row inserted without naming the column explicitly — a seed, a fixture,
any future admin tooling — silently received voting rights. The signup handler
happened to set it explicitly, so the hole was latent rather than live, but a
gate that depends on every caller remembering it isn't a gate.

**Fixed** — defaults to `false` in both `schema.sql` and migration 0003.

### 1.4 Non-idempotent retraction, double-charged penalty (fixed)

`compliance.Retract` read the article on the pool, then opened a *separate*
transaction to write. Two concurrent retract calls could both read
`is_retracted = false` and both apply the FR-15 rank deduction. Calling the
endpoint twice deducted twice, permanently, with no way to reverse it.

**Fixed** — the read moved inside the transaction with `FOR UPDATE`, and an
`is_retracted` re-check makes a repeat call a no-op.

### 1.5 Published JWT secret with no guard (fixed)

`JWT_SECRET` fell back to `dev-secret-change-me`, hardcoded again in
`docker-compose.yml`. Anyone with the repo can forge a token for any user in
any role, including `admin`. Nothing warned about it.

**Fixed** — `config.Validate()` warns in development and **refuses to start**
under `APP_ENV=production` with the default secret, a secret under 32
characters, or `CORS_ALLOW_ORIGINS=*`.

### 1.6 Author signatures are never verified (**open — not fixed**)

`articles.Create` accepts whatever string the client puts in `signature` and
stores it. Nothing verifies it against a key belonging to the journalist. The
frontend generates a throwaway RSA-PSS keypair **per browser session**
(`lib/crypto.ts`), so the signature can't be verified even in principle — the
key is gone once the tab closes.

NFR-4 and F-22 both call for cryptographic author signatures. What exists is a
signature-*shaped* text column. `SRS_FEATURE_STATUS.md` already flagged this as
Partial and that assessment was correct.

Left open deliberately: a real fix needs a keypair registered at signup, a
public key column, and server-side verification on every write — a change to
the signup flow, not a patch. Scoped out of this pass; see §5.

### 1.7 No rate limiting on `/auth/login` (**open — not fixed**)

Unlimited password attempts. bcrypt makes each guess expensive, which is a real
mitigation, but it also makes login a cheap way to burn all your CPU. Needs a
per-IP/per-account limiter.

### 1.8 Infrastructure exposure (**dev-only, documented**)

Every service publishes to `0.0.0.0` — Postgres 5432, Neo4j 7687, Redis 6379
(no password), Kafka 9092. On a laptop behind NAT this is fine; on any shared
network it is not. Credentials are identical across compose, `.env.example`,
and defaults. NFR-5 (HTTPS/TLS) is unimplemented — everything is plaintext
HTTP.

## 2. Correctness

### 2.1 Consensus verdicts were non-deterministic (fixed)

The worst bug found. `EvaluateCrossTagConsensus` ranged over a
`map[bool]map[string]bool` and returned the first side it found holding two
distinct tags:

```go
for verdict, tags := range tagsForVerdict {   // randomised iteration order
    if len(tags) >= 2 { return verdict, true }
}
```

When both sides qualified, **the verdict depended on Go's randomised map
iteration order** — the same votes could resolve `true` on one request and
`false` on the next. A claim's truth value was a coin flip.

**Fixed** — both sides are evaluated explicitly and compared by weight.
`TestEvaluateCrossTagConsensus_IsDeterministic` locks it down.

### 2.2 Only the auditor's first tag was ever read (fixed)

```sql
SELECT u.tags[1], v.verdict FROM votes v JOIN users u ...
```

"Non-overlapping category tags" (FR-7) is a property of whole tag sets. Reading
one tag each made an auditor tagged `{Economic, Labour}` indistinguishable from
one tagged `{Economic}` — so two auditors who both cover Labour were counted as
a valid cross-tag pair when they are not independent at all.

**Fixed** — full tag sets are loaded and genuine set-disjointness is tested.

### 2.3 Resolution wasn't atomic (fixed)

Claim status, article counters, rank score, leaderboard and slashing ran as
separate un-transacted statements. A failure partway left a claim marked
resolved with counters never bumped, or auditors slashed for a verdict the
claim didn't record. Two concurrent voters could both resolve the same claim
and double-bump its article's counters.

**Fixed** — one transaction, with `SELECT ... FOR UPDATE` and a status
re-check.

### 2.4 The CDC consumer died permanently on first error (fixed)

```go
if err != nil { log.Printf(...); return }   // gives up forever
```

On a cold `docker compose up` the backend beats Kafka to readiness, so this
fired essentially every time. One logged line, then **nothing ever synced to
Neo4j again for the process lifetime**. This is exactly what NFR-8 (fault
tolerance, replay on failure) rules out. I saw it live: the first boot logged
`kafka read error` at startup and never recovered.

**Fixed** — capped exponential backoff, retry forever, exit only on context
cancellation.

### 2.5 Lineage edges silently dropped on out-of-order CDC (fixed)

The parent link used `MATCH (parent:Article {id: ...})`. CDC gives no ordering
guarantee, so a child can arrive before its parent — `MATCH` finds nothing, the
`SEQUENCE_OF` edge is never created, and nothing ever retries it. Silent
permanent loss of FR-2 lineage.

**Fixed** — `MERGE` the parent as a stub, filled in when its own event lands.

### 2.6 Louvain never saw most of the graph (fixed)

Two compounding problems:

- `nx.Graph.add_edges_from` only creates nodes present in some edge. Every
  article without a parent — i.e. every standalone story — was **absent from
  the partition entirely** and received no `clusterId`.
- `SEQUENCE_OF` alone is a *forest* (each article has ≤1 parent). Louvain over
  a forest just recovers the individual trees. That is not community detection
  in any useful sense, and F-07's acceptance criterion ("related nodes are
  grouped into logical clusters") was not being met in substance.

`HAS_TAG` — named in SRS §5.1 as one of the two relationships Neo4j governs —
**was never written by anything.** The Cypher for it sat unused in
`packages/database/neo4j/queries.ts`.

**Fixed** — a new claims CDC consumer writes `HAS_TAG`; the worker clusters
over lineage *and* co-tag edges with isolated nodes included, weighted and
seeded for stable cluster ids across polls.

### 2.7 `ApplySlashing` slashed nobody (fixed)

It set `aligned_with_consensus` and returned **every** auditor id as "slashed",
aligned or not. The comment claimed the penalty was "folded into
`ranking.AuditorTrustWeight` the next time each auditor's trust weight is
recalculated" — but **nothing in the codebase ever recalculated it**.
`AuditorTrustWeight` (SRS formula 3) was dead code outside its own unit test.

No auditor was ever penalised for anything. F-13's acceptance criterion failed
silently while the feature was marked Done.

**Fixed** — see §3.

### Requirements with no implementation at all

- **NFR-5 (HTTPS/TLS)** — everything is plaintext HTTP.

- **NFR-2** ("low-latency" CDC propagation) — no latency is defined or

## 4. Structure

- **`CLAUDE.md` describes an entirely different project** (a Java code
  visualizer). It is stale and actively misleading to any agent or contributor
  reading it as project memory.
- **Two schema sources of truth**: `schema.sql` (used by compose init) and
  `migrations/`. Nothing enforces that they agree; they had already drifted on
  `credential_verified`. Both were updated together here, but this will drift
  again — the migrations should be the only source.
- **`packages/database/neo4j/queries.ts` was entirely dead code.** Nothing
  imported it. `UPSERT_ARTICLE` had also drifted out of sync with the Cypher
  actually used in `kafka/consumer.go`.
- **`redis/keys.ts` defines `auditorTrust`**, a key nothing reads or writes.
- **Test coverage** is limited to `ranking`, `consensus`, `auth`. Every HTTP
  handler and all SQL is untested; the frontend has no test runner at all.
- **Neo4j had no constraints applied.** `packages/database/neo4j/schema.cypher`
  declared exactly the right uniqueness constraints — and nothing ever executed
  it, so a live database had none. This is not cosmetic: Neo4j's `MERGE` is
  only atomic with respect to a uniqueness constraint, and the two CDC
  consumers both `MERGE (:Article {id})`. It raced in practice, producing 901
  nodes for 900 articles. Now applied at backend startup by
  `db.EnsureNeo4jConstraints`.

## 5. Deliberately left open

Reported, not fixed, with reasons:

1. **F-22 real signature verification** (§1.6) — needs keypair registration at
   signup and a public-key column; a flow change, not a patch.
2. **Login rate limiting** (§1.7) — needs a shared limiter (Redis is already
   there for it).
3. **TLS / NFR-5** — a deployment concern; nothing to change in app code until
   there's a deployment target.
4. **Appeal resolution** — appeals can be filed and staked, but no endpoint
   resolves one, so a staked score is never returned or forfeited. The SRS
   doesn't specify who adjudicates an appeal or on what basis, so implementing
   it would mean inventing the rule.
5. **CI-3 realtime transport** — no SSE/WebSocket layer.
